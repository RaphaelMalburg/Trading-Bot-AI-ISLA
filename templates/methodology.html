How the Bot Decides to Trade

Each hour, the bot runs a six-step pipeline that ingests market data, computes indicators, asks an ML model for a directional prediction, asks an LLM for a sentiment veto, and — if everything aligns — places an order with explicit stop-loss and take-profit levels.

Pipeline Steps

1Data Ingestion

Pull last 30 days of hourly OHLCV bars for BTC/USD and ETH/USD from the Alpaca Crypto API.

Re-instantiates the client on each retry to clear stale connections (5 retries, exponential backoff).

2Feature Engineering

Compute technical indicators that the ML model uses as input features.

  • RSI(14): momentum / overbought-oversold gauge
  • MACD + signal + diff: trend-following
  • EMA20 / 50 / 200: measured as % distance from price
  • Bollinger Bands: normalized width as volatility proxy
  • ATR(14): typical candle range — sizes our stop-loss
  • OBV: volume-flow vs price
  • Lagged returns & RSI for BTC and ETH (1, 2, 3 hours back)

3Random Forest Prediction

A scikit-learn RandomForestClassifier (100 trees, max depth 10) predicts whether the next hour's close will be higher (1) or lower/flat (0).

Trained on hourly BTC data 2020-2023 with walk-forward validation (first 80% train / last 20% test). The model emits a confidence via predict_proba.

A trade is only considered if confidence > {{ "%.0f"|format(confidence_threshold * 100) }}%.

4Sentiment Veto

The latest 5 BTC news headlines are sent to Gemini 2.5 Flash Lite (via OpenRouter) with a low-temperature prompt asking for a single float in [-1.0, 1.0].

This score acts as a safety filter: even if the ML model is bullish, the trade is vetoed when sentiment falls below {{ "%.1f"|format(sentiment_floor) }}. Bearish news overrides bullish models.

5Position Sizing

Risk-budget approach: cap loss to {{ "%.0f"|format(max_risk * 100) }}% of capital per trade, then back out the position size that puts the stop exactly at that loss.

stop_pct = (ATR × {{ sl_atr_mult }}) / price
position_value = (capital × {{ "%.2f"|format(max_risk) }}) / stop_pct

Clamped to [5% capital, 95% capital]. Alpaca crypto is cash-only — no leverage above 1×.

6Order Execution

If all checks pass:

  1. Market Buy — submit and wait for fill.
  2. Stop-Limit SELL at entry − ATR × {{ sl_atr_mult }} with 0.5% slippage allowance, GTC.
  3. Take-profit target at entry + ATR × {{ tp_atr_mult }} — enforced by a 60-second background watcher (Alpaca crypto allows only one resting SELL).

Why a "Soft" Take-Profit?

Alpaca's crypto API allows only one resting SELL order per position quantity — no OCO, no bracket. We choose to keep the SL on the order book (downside protection is the priority) and enforce TP in code:

  • The hourly cycle checks if last close ≥ TP target.
  • A dedicated 60-second watchdog thread also checks the live price and closes the position if it crosses TP between cycles.
  • If TP fires, the SL order is cancelled and the position is market-closed.

Trade-off: a fast intra-minute spike could still be missed, but crossing the TP for a full minute is enough to trigger the watcher.

Trade Ledger

The application maintains its own SQLite ledger as the source of truth — Alpaca is just the broker:

  • When a BUY fills, the bot writes a row immediately, linked to the run_id that triggered it.
  • An incremental sync runs every cycle, closing rows whose matching SELL has filled on Alpaca.
  • The kill switch closes all open rows on the spot (no waiting for the next sync).
  • Dashboards, stats, and the equity curve all read from this ledger — no Alpaca dependency for historical data.
